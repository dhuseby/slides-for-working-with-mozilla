Review Process

  1. Propose a feature on the mailing list.
  2. Create a bug on bugzilla.
  3. Implement the patches.
  4. Flag for peer review (r?).
    1. UX, security, product, and QA.
  5. Incorporate feedback.
  6. Land the patch.

Bugzilla is the Devil we Know

  • The way things get done.
  • The primary communication tool for coders.
  • Flags: r?, r+, r-, ni?
  • Flags are how you will get UX, Security, and QA reviews.

Open Source is Built in the Open

  • Get community review of feature proposals in mailing lists.
  • Build relationships with module owners.
  • Publish your code in a reviewable repository.
    • Attach patches to Bugzilla bugs for review.
    • Github would be best.
    • Bugzilla understands Github pull request URLs.
